# Module Federation for Shell Composition

We chose Module Federation (via `@originjs/vite-plugin-federation`) to compose the shell host with the three microfrontends at runtime. Each MFE exposes a single React component that the shell mounts inside its layout; routing is owned entirely by the shell using React Router with lazy-loaded remote entries. We rejected **iframe** embedding because it prevents shared styling, breaks deep-linking, and adds cross-origin complexity; **npm packages** because shipping compiled MFEs as packages defeats independent deployment and requires a publish cycle per change; **single-spa** because it introduces a custom lifecycle framework that adds cognitive overhead and a new dependency we would need to maintain; and **Vite SSR / import maps** because neither supports runtime composition of independently deployed SPAs without bundling everything into a single build. Module Federation fits naturally into our existing Vite toolchain, lets each MFE run and deploy independently while sharing a unified UI at runtime, and keeps the shell thin — just layout, routing, and error boundaries. The trade-off is that shared dependencies (React, React DOM) are not deduplicated across MFEs on day one; each MFE bundles its own copy, and deduplication is a planned follow-up optimisation.
